# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this is

A Next.js 15 (App Router) + Supabase app implementing "The Connection
Network" ā€” a multi-vendor, multi-tier referral platform generalizing
PrimeTurf's original "5% + R500" WhatsApp referral scheme. Vendors set
their own reward terms; connectors refer leads and can recruit other
connectors (earning a Tier 2 override on their downline's commissions);
every trust-relevant event is appended to a hash-chained, publicly
verifiable ledger. See `README.md` for the full domain model, route table,
and `docs/` for product strategy (`tcn-program-strategy.md`), the technical
handoff brief (`developer-handoff-guide.md`), and the exact WhatsApp copy
(`whatsapp-funnel-scripts.md`).

There is no automated test suite in this repo.

## Commands

```bash
npm install
npm run dev            # next dev
npm run build           # next build
npm run lint            # next lint
npm run seed:primeturf        # seeds PrimeTurf as the first vendor
npm run seed:unassigned       # seeds the DIRECT-UNASSIGNED connector + `unassigned` vendor bucket used by request-routing/lead-intake fallback
npm run seed:root-connector   # seeds one connector with no upline (see "Growth is invitation-gated" below); requires ROOT_CONNECTOR_WHATSAPP env var
```

Seed scripts run via `tsx` against `.env.local` (`node --env-file=.env.local --import tsx scripts/...`) and talk to the DB through `supabaseAdmin`, not Prisma.

**Database migrations are applied by hand in the Supabase SQL editor**, there is no `prisma migrate` / Supabase CLI workflow wired up. `supabase/setup.sql` is the one-shot, idempotent source of truth (`create ... if not exists` / `drop ... if exists` throughout, safe to re-run) ā€” paste the whole file in for a fresh project. It's generated by concatenating `schema.sql` + every `migration_000N*.sql` file; **whenever you add a new `migration_000N*.sql`, fold the same statements into `setup.sql` too**, or the "one paste sets up everything" property silently breaks (this happened once already ā€” `migration_0006_payouts_unique.sql` shipped without updating `setup.sql`; both are now in sync). `reset_ledger.sql` and `sync_schema.sql` are older one-off scripts, not part of the current setup path.

There's no local Postgres ā€” you need a live Supabase project to run almost anything beyond `next lint`/`tsc`.

## Prisma is installed but not used for queries

`@prisma/client` + `@prisma/adapter-pg` are dependencies and `lib/prisma.ts` exports a configured `PrismaClient`, but **nothing in `app/` or `lib/` imports it** ā€” `prisma/schema.prisma` has no models. All actual reads/writes go through `lib/supabase.ts`'s `supabase` (anon) / `supabaseAdmin` (service role) clients. Prisma exists purely for its CLI/migration tooling potential (`prisma.config.ts` points `DIRECT_URL` ā€” the session-mode pooler port 5432 ā€” at migrations, while `DATABASE_URL`, the transaction-mode pooler port 6543, is what `PrismaClient` would use at runtime). Don't assume a `prisma.model.findX()` call will work anywhere; use `supabaseAdmin.from('table')` instead, matching every existing route.

## Architecture

**Two data-access clients, chosen by trust level**: `lib/supabase.ts` exports `supabase` (anon key, respects RLS ā€” read-only public data like the ledger feed and vendor directory) and `supabaseAdmin` (service role, bypasses RLS ā€” every write, and any read gated by a route's own auth check). API routes almost always use `supabaseAdmin`; RLS policies in `supabase/setup.sql` allow public `select` on `vendors`/`ledger_entries`/`reviews` and service-role-only on everything else, so a route that forgets which client to use fails at the DB layer, not silently.

**Sessions are signed JWTs, not database-backed.** `lib/auth/session.ts` (edge-safe, `jose`-only) issues HS256 JWTs for both admin (`role: admin`) and vendor (`vendorId` + a truncated SHA-256 fingerprint of the vendor's *current* `password_hash`) sessions, keyed off `SESSION_SECRET`. `lib/admin/auth.ts` and `lib/vendor/auth.ts` are thin re-export wrappers kept for import-path stability. The vendor token's password-hash fingerprint means changing a vendor's password invalidates their existing sessions without a server-side session table ā€” but it also means `verifyVendorSession` needs the vendor's *current* hash fetched from the DB before it can validate a cookie (see the two-step pattern in `app/api/referrals/[id]/status/route.ts`: decode the claimed vendor id unsafely via `unsafeDecodeVendorId`, fetch that vendor's hash, then verify). `middleware.ts` only gates `/admin/:path*`; vendor-scoped routes each check their own cookie inline, there's no vendor-side middleware.

**Password hashing has a live migration path.** `lib/auth/password.ts` verifies bcrypt hashes and, for backward compatibility, unsalted-SHA-256 legacy hashes (from before bcrypt was introduced); a successful legacy match returns `needsUpgrade: true` so the caller can transparently re-hash with bcrypt on that login. If you touch vendor login/password-reset code, check whether the caller is honoring `needsUpgrade`.

**The ledger is the trust mechanism, not an afterthought.** Every state change that matters publicly (a connector joining, a referral closing, a commission paid, a review, an eco-pledge honoured, a grade promotion, even an inbound WhatsApp message's routing outcome) is appended via `appendLedgerEntry()` (`lib/ledger/hashChain.ts`), which calls the Postgres function `append_ledger_entry()` ā€” never a direct `insert` into `ledger_entries`. That function holds an advisory lock (`pg_advisory_xact_lock`) so concurrent appends can't fork the hash chain, and computes `hash = sha256(seq : prev_hash : payload_hash)`. `verify_ledger_chain()` (wrapped by `lib/ledger/verify.ts`) recomputes every hash from genesis and is callable with just the anon key ā€” no admin auth needed to prove the log hasn't been tampered with. When adding a new event type, add it to the `entry_type` check constraint in `supabase/setup.sql` *and* the `LedgerEntryType` union in `lib/ledger/types.ts`; they must stay in sync or inserts will fail the DB constraint. Ledger payloads deliberately never carry phone numbers or free-text message content (see the comment in the WhatsApp webhook) ā€” keep that convention for any new entry type.

**The `won` transition is the one write path with real side-effect ordering to get right** (`app/api/referrals/[id]/status/route.ts`): it's guarded against double-fire with a conditional `UPDATE ... WHERE status <> 'won'` (safe under concurrency via Postgres row locking, not just sequential double-clicks) before it calculates commission (`lib/commission/calc.ts`, pure ā€” no DB/DOM), appends `referral_won` + `commission_tier1_paid` ledger entries, inserts a `payouts` row (uniqueness enforced by `migration_0006_payouts_unique.sql` as a DB-level backstop against duplicate billed WhatsApp sends), notifies the connector, runs the grade-promotion check (`lib/connectors/grade.ts`, never throws ā€” a promotion-check failure must not break the payout flow), and ā€” only if there's an upline connector ā€” repeats the tier-2 override flow using either the vendor's default override or the upline's promoted 15% (`overridePctForGrade`) if they're `active_partner`/`ambassador`.

**Growth is invitation-gated by design**, not a bug: `POST /api/connectors` (`app/join`) always requires a valid `uplineReferralCode` ā€” see the comment in `app/api/connectors/route.ts`. There's no public path to becoming the network's first connector; run `npm run seed:root-connector` once per deployment to create a connector with no upline and hand out its code.

**WhatsApp is one inbound funnel with several jobs layered on the same webhook** (`app/api/whatsapp/webhook/route.ts`): Meta's verify handshake (`GET`), HMAC signature verification over the *raw* body via `lib/whatsapp/webhook.ts` (must run before `JSON.parse` ā€” the signature commits to the exact bytes; the endpoint fails closed with 401 if `WHATSAPP_APP_SECRET` is unset, since this is public ledger-writing ingestion), per-message-id idempotency via the `processed_whatsapp_messages` table (Meta redelivers if you don't 200 fast enough), keyword routing (`CONNECT`/`JOIN`/`READY`/`A`/`B`/`C`/`RESET`/`FORGOT`/personalized `JOIN <code>`), and ā€” when a message matches none of that *and* the sender isn't a known connector/vendor ā€” falls through to the MVP request-routing engine (`lib/routing/`): deterministic keyword-based category/location extraction (`detect.ts`, explicitly "keyword lookup, not NLP"), then `findMatchingVendor()` (`match.ts`) which prefers a location match within the same category, else the first active vendor in that category, else the `unassigned` bucket seeded by `seed-unassigned.ts` ā€” a request is never dropped, just routed for manual triage if nothing matches. `sendWhatsAppText` vs `sendWhatsAppTemplate`/`notifyEvent` (`lib/whatsapp/client.ts`) matters: free-form text only delivers within Meta's 24h customer-service window, so business-initiated notifications (referral won, override earned) go through `notifyEvent`, which prefers an approved template (`WHATSAPP_TEMPLATE_*` env vars) and falls back to text.

**Outbound WhatsApp/email calls never throw** (`notify()`, `notifyEvent()` in `lib/whatsapp/client.ts`; `sendEmail()` in `lib/email/client.ts`) and silently no-op when credentials are unset ā€” a notification failure must never break the referral/payout flow that triggered it. Contrast this with the webhook's *inbound* signature check, which fails closed (401) precisely because that's a public write path into the tamper-evident ledger, not a best-effort side effect.

**Shared constants prevent taxonomy drift**: `lib/routing/categories.ts`'s `CATEGORIES` list is the single source used by vendor signup, the admin invitation form, and (implicitly, via `lib/routing/detect.ts`'s keyword map) request capture ā€” add a category in one place if you add it, and update the keyword map too if it should be reachable from a free-text WhatsApp message.
