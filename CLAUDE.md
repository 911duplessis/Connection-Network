# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this is

A Next.js 15 (App Router) + Supabase app implementing "The Connection
Network" — a multi-vendor, multi-tier referral platform generalizing
PrimeTurf's original "5% + R500" WhatsApp referral scheme. Vendors set
their own reward terms; connectors refer leads and can recruit other
connectors (earning a Tier 2 override on their downline's commissions);
every trust-relevant event is appended to a hash-chained, publicly
verifiable ledger. See `README.md` for the full domain model, route table,
and `docs/` for product strategy (`tcn-program-strategy.md`), the technical
handoff brief (`developer-handoff-guide.md`), and the exact WhatsApp copy
(`whatsapp-funnel-scripts.md`).

There is no automated test suite in this repo.

## Commands

```bash
npm install
npm run dev            # next dev
npm run build           # next build
npm run lint            # next lint
npm run seed:primeturf        # seeds PrimeTurf as the first vendor
npm run seed:unassigned       # seeds the DIRECT-UNASSIGNED connector + `unassigned` vendor bucket used by request-routing/lead-intake fallback
npm run seed:root-connector   # seeds one connector with no upline (see "Growth is invitation-gated" below); requires ROOT_CONNECTOR_WHATSAPP env var
```

Seed scripts run via `tsx` against `.env.local` (`node --env-file=.env.local --import tsx scripts/...`) and talk to the DB through `supabaseAdmin`, not Prisma.

**Database migrations are applied by hand in the Supabase SQL editor**, there is no `prisma migrate` / Supabase CLI workflow wired up. `supabase/setup.sql` is the one-shot, idempotent source of truth (`create ... if not exists` / `drop ... if exists` throughout, safe to re-run) — paste the whole file in for a fresh project. It's generated by concatenating `schema.sql` + every `migration_000N*.sql` file; **whenever you add a new `migration_000N*.sql`, fold the same statements into `setup.sql` too**, or the "one paste sets up everything" property silently breaks (this happened once already — `migration_0006_payouts_unique.sql` shipped without updating `setup.sql`; both are now in sync). `reset_ledger.sql` and `sync_schema.sql` are older one-off scripts, not part of the current setup path.

There's no local Postgres — you need a live Supabase project to run almost anything beyond `next lint`/`tsc`.

## Prisma is installed but not used for queries

`@prisma/client` + `@prisma/adapter-pg` are dependencies and `lib/prisma.ts` exports a configured `PrismaClient`, but **nothing in `app/` or `lib/` imports it** — `prisma/schema.prisma` has no models. All actual reads/writes go through `lib/supabase.ts`'s `supabase` (anon) / `supabaseAdmin` (service role) clients. Prisma exists purely for its CLI/migration tooling potential (`prisma.config.ts` points `DIRECT_URL` — the session-mode pooler port 5432 — at migrations, while `DATABASE_URL`, the transaction-mode pooler port 6543, is what `PrismaClient` would use at runtime). Don't assume a `prisma.model.findX()` call will work anywhere; use `supabaseAdmin.from('table')` instead, matching every existing route.

## Architecture

**Two data-access clients, chosen by trust level**: `lib/supabase.ts` exports `supabase` (anon key, respects RLS — read-only public data like the ledger feed and vendor directory) and `supabaseAdmin` (service role, bypasses RLS — every write, and any read gated by a route's own auth check). API routes almost always use `supabaseAdmin`; RLS policies in `supabase/setup.sql` allow public `select` on `vendors`/`ledger_entries`/`reviews` and service-role-only on everything else, so a route that forgets which client to use fails at the DB layer, not silently.

**Sessions are signed JWTs, not database-backed.** `lib/auth/session.ts` (edge-safe, `jose`-only) issues HS256 JWTs for both admin (`role: admin`) and vendor (`vendorId` + a truncated SHA-256 fingerprint of the vendor's *current* `password_hash`) sessions, keyed off `SESSION_SECRET`. `lib/admin/auth.ts` and `lib/vendor/auth.ts` are thin re-export wrappers kept for import-path stability. The vendor token's password-hash fingerprint means changing a vendor's password invalidates their existing sessions without a server-side session table — but it also means `verifyVendorSession` needs the vendor's *current* hash fetched from the DB before it can validate a cookie (see the two-step pattern in `app/api/referrals/[id]/status/route.ts`: decode the claimed vendor id unsafely via `unsafeDecodeVendorId`, fetch that vendor's hash, then verify). `middleware.ts` only gates `/admin/:path*`; vendor-scoped routes each check their own cookie inline, there's no vendor-side middleware.

**A second auth path bridges into Supabase Auth for live dashboard updates only — it is not a general-purpose session system.** Production has a `user_roles` table (`auth.uid()` → `vendor_id`/`connector_id`), owner-scoped RLS read policies, and Realtime Authorization on `realtime.messages` (private channels `connector:<id>:dashboard` / `vendor:<id>:dashboard`) — see `supabase/migration_0008_realtime_dashboard_auth.sql` and `supabase/migration_0009_bridge_identity_unique.sql` (partial unique indexes on `user_roles.vendor_id`/`connector_id`, since GoTrue's own email uniqueness dedups `auth.users` but nothing enforced "at most one identity per vendor/connector" in `user_roles` itself). `lib/auth/bridge.ts`'s `createBridgeSession(role, id)` mints a minimal session on demand: `supabaseAdmin.auth.admin.generateLink({type:'magiclink', email: bridgeEmailFor(role,id)})` against a deterministic synthetic email (`${role}+${id}@bridge.tcn.internal`, no real mail ever sent — `generateLink` is already idempotent find-or-create, no separate `createUser` call needed) — then `verifyOtp({token_hash, type:'magiclink'})` on a **throwaway** anon client (never the shared `lib/supabase.ts` `supabase` export, which is a reused singleton — running `verifyOtp` on it would leak session state across concurrent requests) to get an `access_token`. Only that access token is ever returned to the browser (no `refresh_token`, no `setSession()`) — the client calls `realtime.setAuth(accessToken)` (see `lib/supabase/browserClient.ts`, `components/LiveRefresh.tsx`) purely to authorize the private-channel WebSocket handshake, nothing else reads/writes through it. `app/api/vendor/bridge-session` and `app/api/connector/bridge-session` gate minting behind the *same* credentials each dashboard already requires (vendor session cookie / connector whatsapp+referralCode) — this bridge never weakens the existing auth model, it rides on top of it. Server-side broadcasts (`lib/realtime/broadcast.ts`'s `broadcastDashboardEvent()`, wired into referral creation and the status-transition route) are bare "something changed, refetch" signals sent via `supabaseAdmin` (service role bypasses RLS; `realtime.messages` has no `insert` policy for `anon`/`authenticated`, so clients can never send, only receive) — never a data stream, never a ledger-worthy event.

**Password hashing has a live migration path.** `lib/auth/password.ts` verifies bcrypt hashes and, for backward compatibility, unsalted-SHA-256 legacy hashes (from before bcrypt was introduced); a successful legacy match returns `needsUpgrade: true` so the caller can transparently re-hash with bcrypt on that login. If you touch vendor login/password-reset code, check whether the caller is honoring `needsUpgrade`.

**The ledger is the trust mechanism, not an afterthought.** Every state change that matters publicly (a connector joining, a referral closing, a commission paid, a review, an eco-pledge honoured, a grade promotion, even an inbound WhatsApp message's routing outcome) is appended via `appendLedgerEntry()` (`lib/ledger/hashChain.ts`), which calls the Postgres function `append_ledger_entry()` — never a direct `insert` into `ledger_entries`. That function holds an advisory lock (`pg_advisory_xact_lock`) so concurrent appends can't fork the hash chain, and computes `hash = sha256(seq : prev_hash : payload_hash)`. `verify_ledger_chain()` (wrapped by `lib/ledger/verify.ts`) recomputes every hash from genesis and is callable with just the anon key — no admin auth needed to prove the log hasn't been tampered with. When adding a new event type, add it to the `entry_type` check constraint in `supabase/setup.sql` *and* the `LedgerEntryType` union in `lib/ledger/types.ts`; they must stay in sync or inserts will fail the DB constraint. Ledger payloads deliberately never carry phone numbers or free-text message content (see the comment in the WhatsApp webhook) — keep that convention for any new entry type.

**The `won` transition is the one write path with real side-effect ordering to get right** (`app/api/referrals/[id]/status/route.ts`): it's guarded against double-fire with a conditional `UPDATE ... WHERE status <> 'won'` (safe under concurrency via Postgres row locking, not just sequential double-clicks) before it calculates commission (`lib/commission/calc.ts`, pure — no DB/DOM), appends `referral_won` + `commission_tier1_paid` ledger entries, inserts a `payouts` row (uniqueness enforced by `migration_0006_payouts_unique.sql` as a DB-level backstop against duplicate billed WhatsApp sends), notifies the connector, runs the grade-promotion check (`lib/connectors/grade.ts`, never throws — a promotion-check failure must not break the payout flow), and — only if there's an upline connector — repeats the tier-2 override flow using either the vendor's default override or the upline's promoted 15% (`overridePctForGrade`) if they're `active_partner`/`ambassador`.

**Every referral status transition is logged and reported, not just `won`.** The same route appends a `referral_status_changed` ledger entry (`{referralId, vendorSlug, connectorId, fromStatus, toStatus}`) and notifies the connector via `notifyEvent()` for `contacted`/`quoted`/`lost` too — this closes what was previously a one-sided flow where a connector only heard from the vendor once a deal paid out. `components/ReferralRow.tsx` frames the vendor's first move on a `submitted` referral as explicit **Accept** (→ `contacted`) / **Decline** (→ `lost`) buttons rather than a generic status dropdown; the dropdown only appears once a referral is past `submitted`, and can no longer move a referral backward to `submitted`. A separate `connector_invited` ledger entry type backs the "invite this lead to become a connector" action (`app/api/referrals/[id]/invite-connector/route.ts`) available on `whatsapp_request`-sourced referrals (leads with no real connector attached, bucketed under `UNASSIGNED_CONNECTOR_CODE`) — it WhatsApps the lead a `/join?upline=DIRECT-UNASSIGNED` link and is idempotent via `referrals.connector_invite_sent_at`. A vendor becoming a connector themselves needs no new endpoint: the vendor dashboard's "Become a connector too" link reuses the same `/join?upline=...` prefill mechanism (`app/join/page.tsx` reads `upline`/`name`/`whatsapp`/`email` query params into the existing form, still gated on the same agreement checkbox and `POST /api/connectors` call as any other signup).

**The financial lifecycle of a referral has three distinct numbers, not one.** `referrals.quoted_value_cents` (set optionally on the `quoted` transition, via the same `window.prompt` pattern `won` already used for `job_value_cents`) is the proposed value; `job_value_cents` at `won` is the agreed/final value; `payouts.amount_cents` is the commission calculated from it. A fourth state, `payouts.paid_at`, answers a question the schema couldn't answer before: not "was a commission calculated" but "was it actually disbursed" — set via `POST /api/admin/payouts/:id/mark-paid` (admin-only, idempotent — a second call on an already-paid payout is a no-op) and logged as a `payout_marked_paid` ledger entry, since "this connector was actually paid" is exactly the kind of event this platform exists to make provable. `app/admin/page.tsx` surfaces all of this as an overview panel (referral counts by stage, quoted/won totals, commissions owed vs. paid, eco-pledge accrued — all grouped by vendor currency, derived server-side from `referrals`/`payouts`/`ledger_entries` on every page load, not a separate materialized/cached stat) plus a dedicated Payouts table with the mark-paid action, in addition to the existing Referrals table.

**Growth is invitation-gated by design**, not a bug: `POST /api/connectors` (`app/join`) always requires a valid `uplineReferralCode` — see the comment in `app/api/connectors/route.ts`. There's no public path to becoming the network's first connector; run `npm run seed:root-connector` once per deployment to create a connector with no upline and hand out its code.

**WhatsApp is one inbound funnel with several jobs layered on the same webhook** (`app/api/whatsapp/webhook/route.ts`): Meta's verify handshake (`GET`), HMAC signature verification over the *raw* body via `lib/whatsapp/webhook.ts` (must run before `JSON.parse` — the signature commits to the exact bytes; the endpoint fails closed with 401 if `WHATSAPP_APP_SECRET` is unset, since this is public ledger-writing ingestion), per-message-id idempotency via the `processed_whatsapp_messages` table (Meta redelivers if you don't 200 fast enough), keyword routing (`CONNECT`/`JOIN`/`READY`/`A`/`B`/`C`/`RESET`/`FORGOT`/personalized `JOIN <code>`), and — when a message matches none of that *and* the sender isn't a known connector/vendor — falls through to the MVP request-routing engine (`lib/routing/`): deterministic keyword-based category/location extraction (`detect.ts`, explicitly "keyword lookup, not NLP"), then `findMatchingVendor()` (`match.ts`) which prefers a location match within the same category, else the first active vendor in that category, else the `unassigned` bucket seeded by `seed-unassigned.ts` — a request is never dropped, just routed for manual triage if nothing matches. `sendWhatsAppText` vs `sendWhatsAppTemplate`/`notifyEvent` (`lib/whatsapp/client.ts`) matters: free-form text only delivers within Meta's 24h customer-service window, so business-initiated notifications (referral won, override earned) go through `notifyEvent`, which prefers an approved template (`WHATSAPP_TEMPLATE_*` env vars) and falls back to text.

**Outbound WhatsApp/email calls never throw** (`notify()`, `notifyEvent()` in `lib/whatsapp/client.ts`; `sendEmail()` in `lib/email/client.ts`) and silently no-op when credentials are unset — a notification failure must never break the referral/payout flow that triggered it. Contrast this with the webhook's *inbound* signature check, which fails closed (401) precisely because that's a public write path into the tamper-evident ledger, not a best-effort side effect.

**Shared constants prevent taxonomy drift**: `lib/routing/categories.ts`'s `CATEGORIES` list is the single source used by vendor signup, the admin invitation form, and (implicitly, via `lib/routing/detect.ts`'s keyword map) request capture — add a category in one place if you add it, and update the keyword map too if it should be reachable from a free-text WhatsApp message.

## Current status (as of the architecture-audit + hardening pass)

**Functional and verified live in production**, not just code-reviewed: the full referral lifecycle (submit via web form, WhatsApp free-text, or the public lead form → vendor Accept/Decline → quoted → won/lost), atomic commission writes (`process_won_commissions()`, migration_0011), the tamper-evident ledger (hash chain verified clean via `/api/ledger/verify`), admin login and vendor login (both were silently broken — `SESSION_SECRET` had never been set in production until this pass), the admin overview panel (referral counts, quoted/won totals, commissions owed vs. paid, eco-pledge accrued), and payout paid-tracking. Migrations 0010–0012 are applied in production; `applied_migrations` now tracks that going forward instead of it being tribal knowledge.

**Known limitation, explicitly not yet addressed**: the vendor and connector dashboards are still the original flat referral-row / earnings-total views — they didn't get the overview treatment the admin dashboard just got (referral counts by stage, quoted vs. won breakdown, payout status). Worth doing before those get scoped as separate follow-up work, since it's the same pattern already built for admin.

**Explicitly out of scope for now, by design**: any real payment rail (smart contracts, escrow, or otherwise). `payouts.paid_at` is a manually-recorded admin attestation ("I paid this connector via EFT/cash/whatever"), not a real money movement — there is no payment processor integration anywhere in this codebase. That's a genuinely separate, later initiative, not a small addition on top of what exists.

**Still-open operational item, unrelated to the app code**: `WHATSAPP_ACCESS_TOKEN` is a short-lived Graph API Explorer token (~2hr lifetime) that has now expired and been manually replaced multiple times this session. The standing fix — a permanent Meta System User access token — hasn't been done yet.
